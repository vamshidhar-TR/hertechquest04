🧭 Voice Pipeline Architecture

The reference shape for a hands-free voice → AI → answer app. Each stage maps to one of the other tech docs.

The flow

        ┌───────── on your machine (audio stays local) ─────────┐
        │                                                       │

One seam to remember. Everything left of stage 4 is the voice layer (given to you in starter/). Everything right of it is your backend, reached through a single function — answer(question). Swap what's behind it (CoCounsel over MCP, a REST API, a local model) without touching the voice UI.

Design choices that matter

Audio stays local. Capture + transcription run on-device; only the question text leaves the machine. Keeps client/privileged speech private.

Push-to-talk, not always-on. Capture only when asked, so stray speech and phone calls don't get sent to the backend.

Transcription on background threads. Inference is slow relative to the UI — run capture, transcription, and the backend call off the UI thread and post results back, or the window freezes.

The answer() seam is the whole point. The UI only knows answer(question) → text. That indirection is what lets every team point the same app at a different backend.

Minimal contract between stages

audio:    np.float32 mono, 16 kHz, range [-1, 1]
   ▼ transcribe(audio) -> str
question: plain text (user may edit)
   ▼ answer(question) -> str        # markdown links become clickable sources
answer:   text (+ optional [label](https://…) links)
